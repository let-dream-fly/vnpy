## 1. 项目概述

VeighNa（原 vn.py）是一套基于 Python 的开源量化交易系统开发框架，历经十年社区积累演进为功能完备的多功能量化交易平台。4.0 版本起引入 AI 量化模块，定位为：

> **By Traders, For Traders, AI-Powered**

**核心价值主张**

| 维度 | 描述 |
|------|------|
| 全市场覆盖 | 国内外期货、股票、期权、数字货币等品种的统一接口 |
| 全流程支持 | 数据获取 → 因子挖掘 → 策略回测 → 实盘交易一体化 |
| 高扩展性 | 网关、数据库、数据源、策略应用均采用插件架构 |
| AI 赋能 | 内置多因子机器学习工作流（ML策略、特征工程、模型训练） |
| 跨平台 | Windows / Linux / macOS，Python 3.10–3.13 |

**版本历史里程碑**

| 版本 | 关键特性 |
|------|---------|
| 2.0 | 统一网关抽象，事件驱动架构定型 |
| 3.0 | 插件包体系（`vnpy_*`），模块化解耦 |
| 4.0 | AI 量化模块（`vnpy.alpha`），PySide6 升级 |
| 4.3 | VeighNa Assistant（AI 智能助手），Alpha 101 因子集，Docker 镜像优化 |

**架构核心思想**

整个平台以**事件总线（EventEngine）为脊梁**，遵循单向数据流原则：

- **请求向下**：策略 → 主引擎 → 网关 → 交易所
- **数据向上**：交易所 → 网关回调 → 事件引擎 → OMS / 策略 / UI

所有插件（网关、应用、数据库、数据源）通过统一抽象基类接入，形成可无限横向扩展的生态体系。

## 3. 核心框架层（vnpy.trader）

### 3.1 事件引擎（`vnpy/event/engine.py`）

事件引擎是整个平台的**消息总线**，采用生产者-消费者模式实现进程内异步解耦。

**源码核心结构**

```python
class EventEngine:
    def __init__(self, interval: int = 1) -> None:
        self._queue: Queue = Queue()          # 线程安全事件队列
        self._active: bool = False
        self._thread: Thread = Thread(target=self._run)    # 事件分发线程
        self._timer: Thread = Thread(target=self._run_timer)  # 定时器线程
        self._handlers: defaultdict = defaultdict(list)    # 精确事件订阅
        self._general_handlers: list = []                  # 全局事件订阅

    def _process(self, event: Event) -> None:
        # 精确分发：只通知订阅了该事件类型的 handler
        if event.type in self._handlers:
            [handler(event) for handler in self._handlers[event.type]]
        # 全局分发：通知所有通用 handler（UI 刷新等）
        if self._general_handlers:
            [handler(event) for handler in self._general_handlers]
```

**核心数据结构**

```
Event
├── type: str          # 事件类型字符串（路由键）
└── data: Any          # 载荷（任意Python对象）
```

**工作机制**

```
生产者（网关/引擎）
    │  put(event)
    ▼
  Queue（线程安全）
    │
    ▼  worker thread（_run）
  _process(event)
    ├── 分发给 handlers[event.type]   # 精确订阅
    └── 分发给 general_handlers       # 全局订阅（UI等）
    
  Timer thread（_run_timer，每 interval 秒）
    └── put(Event(EVENT_TIMER))       # 定时心跳，驱动轮询逻辑
```

**关键事件类型**

| 常量 | 描述 |
|------|------|
| `EVENT_TICK` | 行情 Tick 数据 |
| `EVENT_ORDER` | 委托状态更新 |
| `EVENT_TRADE` | 成交回报 |
| `EVENT_POSITION` | 持仓更新 |
| `EVENT_ACCOUNT` | 账户资金更新 |
| `EVENT_CONTRACT` | 合约信息 |
| `EVENT_LOG` | 日志事件 |
| `EVENT_TIMER` | 定时器事件（每秒触发） |
| `EVENT_QUOTE` | 做市报价事件 |

**双频道事件命名约定**

网关在 `on_tick`、`on_order` 等方法中**同时推送两个事件**：
- 全局事件：`EVENT_TICK`（OMS 和所有监听者接收）
- 精确事件：`EVENT_TICK + vt_symbol`（只订阅特定合约的策略接收）

这种设计使 OMS 做全局状态聚合，而策略只响应自己关心的品种，避免无效回调。

**设计模式**：发布-订阅 + 生产者-消费者 + 观察者

### 3.2 主引擎（`MainEngine`）

`MainEngine` 是整个平台的**组合根（Composition Root）**，承担插件注册、路由分发和生命周期管理。

**初始化流程**

```python
MainEngine(event_engine)
    ├── 持有 EventEngine，调用 event_engine.start()
    ├── chdir(TRADER_DIR)            # 切换工作目录到 .vntrader/
    └── init_engines()
            ├── add_engine(LogEngine)
            ├── add_engine(OmsEngine)    # 并将 OMS 查询方法绑定到 MainEngine
            └── add_engine(EmailEngine)
```

**插件注册 API**

| 方法 | 说明 |
|------|------|
| `add_gateway(gateway_class)` | 注册交易网关，存入 `gateways` 字典 |
| `add_engine(engine_class)` | 注册功能引擎，存入 `engines` 字典 |
| `add_app(app_class)` | 注册应用模块（自动添加其 engine_class），返回 engine 实例 |

**交易路由 API**（委托给具名网关）

```python
main_engine.connect(setting, gateway_name)         # 建立连接
main_engine.subscribe(req, gateway_name)           # 订阅行情
main_engine.send_order(req, gateway_name)          # → 返回 vt_orderid
main_engine.cancel_order(req, gateway_name)        # 撤单
main_engine.send_quote(req, gateway_name)          # 做市报价
main_engine.query_history(req, gateway_name)       # → List[BarData]
```

**设计模式**：外观模式（Facade）+ 单例中心节点

### 3.3 订单管理系统（`OmsEngine`）

`OmsEngine` 是纯内存的**状态管理器**，监听所有市场与交易事件，维护平台运行期间的完整状态快照。

**维护的数据集合**

| 集合 | 键 | 值类型 |
|------|----|--------|
| `ticks` | `vt_symbol` | `TickData` |
| `orders` | `vt_orderid` | `OrderData` |
| `active_orders` | `vt_orderid` | `OrderData`（仅活动委托）|
| `trades` | `vt_tradeid` | `TradeData` |
| `positions` | `vt_positionid` | `PositionData` |
| `accounts` | `vt_accountid` | `AccountData` |
| `contracts` | `vt_symbol` | `ContractData` |
| `quotes` | `vt_quoteid` | `QuoteData` |
| `offset_converters` | `gateway_name` | `OffsetConverter` |

**事件处理链**

```
EVENT_CONTRACT → 创建该网关的 OffsetConverter
EVENT_TICK     → 更新 ticks[vt_symbol]
EVENT_ORDER    → 更新 orders，维护 active_orders；喂给 OffsetConverter
EVENT_TRADE    → 更新 trades；喂给 OffsetConverter（更新持仓冻结量）
EVENT_POSITION → 更新 positions；喂给 OffsetConverter
EVENT_ACCOUNT  → 更新 accounts
EVENT_QUOTE    → 更新 quotes，维护 active_quotes
```

OmsEngine 的查询方法被绑定到 MainEngine 上，策略和 UI 可直接通过 `main_engine.get_tick(vt_symbol)` 等方法获取全局状态。

### 3.4 网关接口（`BaseGateway`）

网关是交易所/券商 API 与 VeighNa 统一模型之间的**适配器**，每个具体网关对应一个独立的 `vnpy_*` 包。

**类级元数据**（子类必须声明）

```python
class CtpGateway(BaseGateway):
    default_name = "CTP"
    default_setting = {
        "用户名": "",
        "密码": "",
        "经纪商代码": "",
        "交易服务器": "",
        "行情服务器": "",
        ...
    }
    exchanges = [Exchange.SHFE, Exchange.DCE, Exchange.CZCE, ...]
```

**必须实现的抽象方法**

| 方法 | 方向 | 说明 |
|------|------|------|
| `connect(setting)` | 下行 | 建立连接，登录 |
| `close()` | 下行 | 断开连接 |
| `subscribe(req)` | 下行 | 订阅行情 |
| `send_order(req)` | 下行 | 发送委托，返回 orderid |
| `cancel_order(req)` | 下行 | 撤销委托 |
| `query_account()` | 下行 | 查询账户资金 |
| `query_position()` | 下行 | 查询持仓 |

**事件回调方法**（已实现，网关内部调用，推送事件上行）

```python
self.on_tick(tick)       # → EVENT_TICK + EVENT_TICK.vt_symbol
self.on_order(order)     # → EVENT_ORDER + EVENT_ORDER.vt_orderid
self.on_trade(trade)     # → EVENT_TRADE + EVENT_TRADE.vt_tradeid
self.on_position(pos)    # → EVENT_POSITION
self.on_account(acc)     # → EVENT_ACCOUNT
self.on_contract(con)    # → EVENT_CONTRACT + EVENT_CONTRACT.vt_symbol
```

**设计模式**：模板方法（Template Method）+ 适配器（Adapter）

## 4. 基础设施层

### 4.1 数据库适配器（`vnpy/trader/database.py`）

采用**抽象基类 + 插件包**的设计，所有数据库实现发布为独立 `vnpy_<name>` 包。

**统一接口（`BaseDatabase`）**

```python
class BaseDatabase(ABC):
    # 写入
    def save_bar_data(self, bars: List[BarData], stream: bool = False) -> bool: ...
    def save_tick_data(self, ticks: List[TickData], stream: bool = False) -> bool: ...
    # 读取
    def load_bar_data(self, symbol, exchange, interval, start, end) -> List[BarData]: ...
    def load_tick_data(self, symbol, exchange, start, end) -> List[TickData]: ...
    # 删除
    def delete_bar_data(self, symbol, exchange, interval) -> int: ...
    def delete_tick_data(self, symbol, exchange) -> int: ...
    # 概览
    def get_bar_overview(self) -> List[BarOverview]: ...
    def get_tick_overview(self) -> List[TickOverview]: ...
```

**支持的数据库**

| 类型 | 驱动包 | 特点 |
|------|--------|------|
| SQLite | `vnpy_sqlite` | 默认，零配置，适合入门 |
| MySQL | `vnpy_mysql` | 主流关系型，可替换 TiDB |
| PostgreSQL | `vnpy_postgresql` | 功能丰富，支持插件扩展 |
| DolphinDB | `vnpy_dolphindb` | 高性能时序，低延迟场景 |
| TDengine | `vnpy_taos` | 分布式时序，内置流计算 |
| MongoDB | `vnpy_mongodb` | 文档型，热数据内存缓存 |

**时区处理**：统一通过 `DB_TZ = SETTINGS["database.timezone"]` 做 normalize，使用 `tzlocal` 自动获取本地时区。

**动态加载机制**

```python
def get_database() -> BaseDatabase:
    name = SETTINGS["database.name"]              # 读取配置
    module = importlib.import_module(f"vnpy_{name}")  # 动态导入插件包
    return module.database                         # 返回单例实例
```

### 4.5 RPC 跨进程通信（`vnpy/rpc/`）

基于 **ZeroMQ (pyzmq)** 实现分布式部署能力，支持一个服务端对应多个客户端。

**协议设计**

```
REQ-REP 通道（同步命令调用）
  Client                          Server
  send([name, args, kwargs])  →   dispatch(name, *args, **kwargs)
                              ←   send([True, result])
                              ←   send([False, traceback_str])

PUB-SUB 通道（异步数据推送）
  Server                          Client(s)
  publish(topic, data)        →   callback(topic, data)
  publish(HEARTBEAT_TOPIC)    →   liveness check
```

**服务端（`RpcServer`）**

```python
server = RpcServer()
server.register(main_engine.get_tick)     # 注册可远程调用的函数
server.register(main_engine.send_order)
server.start(rep_address="tcp://*:2014", pub_address="tcp://*:4102")
```

**客户端（`RpcClient`）**

```python
client = MyRpcClient()
client.start(req_address="tcp://server:2014", sub_address="tcp://server:4102")
tick = client.get_tick(vt_symbol)          # 魔术方法 __getattr__ 实现透明远程调用
```

**心跳检测**：服务端按 `HEARTBEAT_INTERVAL` 定时推送心跳；客户端在 `HEARTBEAT_TOLERANCE` 内未收到则触发 `on_disconnected()`。

**典型部署场景**：`vnpy_rpcservice` 应用将完整的行情+交易引擎暴露为服务端，多个策略进程作为客户端连接，实现进程级别的策略隔离。

## 5. 工具与算法层

### 5.1 行情处理工具（`vnpy/trader/utility.py`）

**`BarGenerator` — 实时行情合成器**

```
Tick 流
  │  update_tick()
  ▼
1分钟 Bar（on_bar 回调）
  │  update_bar()
  ▼
窗口 Bar（on_window_bar 回调）
  支持：N分钟、N小时、日线（需指定 daily_end 收盘时间）
```

关键细节：
- 分钟 Bar 的 high/low 来自 Tick 的 high/low 字段（而非 last_price），避免 last 价格遗漏盘中极值
- Volume/turnover 使用增量（当前 Tick 减前一 Tick），防止重复计数
- 多分钟窗口：要求 window 能整除 60，通过 `(bar.datetime.minute + 1) % window` 判断完成

**`ArrayManager` — 技术指标计算器**

固定大小的 numpy 环形缓冲区，包装 TA-Lib 实现：

```python
am = ArrayManager(size=100)
am.update_bar(bar)

# 计算指标（返回最新值或完整数组）
sma_10 = am.sma(10)                    # → float
macd, signal, hist = am.macd(12, 26, 9, array=True)  # → ndarray
upper, mid, lower = am.boll(20, 2.0)
rsi_14 = am.rsi(14)
```

支持的指标：SMA、EMA、WMA、DEMA、TEMA、RSI、MACD、Bollinger Bands、KDJ、ATR、CCI、ADXR、SAR、TRIX 等（全部委托 TA-Lib）。

`inited` 标志：当历史数据积累满 `size` 条后置为 `True`，策略以此判断指标是否可靠。

### 5.3 持仓转换器（`vnpy/trader/converter.py`）

专门处理**中国期货市场**的开平仓规则，自动将净头寸委托转换为符合交易所要求的开/平委托组合。

**核心逻辑（`PositionHolding`）**

```
持仓状态
├── long_pos / short_pos          # 总持仓
├── long_td / long_yd             # 多头今仓 / 昨仓（上期所/能源所区分）
├── short_td / short_yd           # 空头今仓 / 昨仓
└── long_pos_frozen / short_pos_frozen  # 活动委托冻结量
```

**转换模式**

| 模式 | 适用场景 | 逻辑 |
|------|----------|------|
| SHFE/INE 模式 | 上期所/上期能源所 | 优先平今（close today first），再平昨，可能拆单 |
| 锁仓模式（lock） | 需要持有双向持仓 | 今仓有持仓则反向开仓锁仓，否则平昨 |
| 净持仓模式（net） | 净持仓合约 | 跳过转换，直接透传 |

净持仓合约（`ContractData.net_position = True`）自动跳过转换。

## 7. 自动交易实现原理

本章深入讲解 VeighNa 如何将行情数据、策略逻辑、委托执行三个环节串联为完整的**自动量化交易流水线**。

### 7.1 CTA 策略完整生命周期

CTA 策略（`CtaTemplate`）的完整生命周期分为以下阶段：

```
1. 注册阶段（引擎初始化时）
   CtaEngine.load_strategy()
       ├── 读取策略配置文件（.vntrader/cta_strategy.json）
       ├── 实例化策略类，绑定 vt_symbol
       └── 调用 strategy.on_init() → 加载历史数据预热指标

2. 启动阶段
   CtaEngine.start_strategy()
       ├── 订阅合约行情：main_engine.subscribe(req, gateway_name)
       ├── 注册精确事件：event_engine.register(EVENT_TICK + vt_symbol, strategy.on_tick)
       └── 调用 strategy.on_start() → 策略就绪，开始接收实时行情

3. 运行阶段（事件驱动循环）
   行情到达 → on_tick() / on_bar() → 信号生成 → 下单 → 成交回报 → 持仓更新

4. 停止阶段
   CtaEngine.stop_strategy()
       ├── 调用 strategy.on_stop()
       ├── 撤销所有活动委托
       └── 取消事件订阅

5. 参数持久化
   strategy.put_event()          # 推送策略状态更新到 UI
   cta_engine.sync_strategy_data() # 保存策略变量到 .vntrader/
```

**历史数据预热（关键细节）**

```python
def on_init(self):
    # 加载 N 天历史 Bar，驱动 on_bar() 预热 ArrayManager
    self.load_bar(10)
    # 此时 am.inited 将变为 True，策略指标已就绪
```

`load_bar()` 从数据库或数据服务拉取历史数据，**模拟驱动** `on_bar()` 回调（不产生真实委托），使技术指标在实盘开始前已经计算出可靠值。

### 7.2 行情驱动的信号生成机制

**Tick → Bar → 指标 → 信号** 是 CTA 策略的标准数据处理链：

```python
class DoubleMaStrategy(CtaTemplate):
    fast_window = Parameter(10)
    slow_window = Parameter(30)

    def on_init(self):
        # BarGenerator：将 Tick 合成 1 分钟 Bar，再合成 15 分钟 Bar
        self.bg = BarGenerator(self.on_bar, 15, self.on_15min_bar)
        self.am = ArrayManager()
        self.load_bar(10)

    def on_tick(self, tick: TickData):
        self.bg.update_tick(tick)      # Tick 流入 → 自动合成 Bar

    def on_bar(self, bar: BarData):
        self.bg.update_bar(bar)        # 1分钟 Bar → 合成 15分钟 Bar

    def on_15min_bar(self, bar: BarData):
        self.cancel_all()              # 先撤销上一轮所有活动委托
        am = self.am
        am.update_bar(bar)
        if not am.inited:
            return

        fast_ma = am.sma(self.fast_window)
        slow_ma = am.sma(self.slow_window)

        # 金叉信号 → 开多
        if fast_ma > slow_ma and self.pos <= 0:
            if self.pos < 0:
                self.cover(bar.close_price, abs(self.pos))  # 平空
            self.buy(bar.close_price, 1)                    # 开多

        # 死叉信号 → 开空
        elif fast_ma < slow_ma and self.pos >= 0:
            if self.pos > 0:
                self.sell(bar.close_price, abs(self.pos))   # 平多
            self.short(bar.close_price, 1)                  # 开空

        self.put_event()               # 刷新 UI 策略状态面板
```

**四个核心交易指令**

| 方法 | 方向 | 开平 | 说明 |
|------|------|------|------|
| `buy(price, volume)` | 多 | 开 | 买入开多 |
| `sell(price, volume)` | 多 | 平 | 卖出平多 |
| `short(price, volume)` | 空 | 开 | 卖出开空 |
| `cover(price, volume)` | 空 | 平 | 买入平空 |

这四个方法内部调用 `CtaEngine.send_order()`，自动经过 **OffsetConverter** 处理开平仓转换，最终调用 `main_engine.send_order()` 发往网关。

### 7.4 持仓状态管理与同步

**`pos` 字段的维护逻辑**

CTA 策略通过 `self.pos` 跟踪净持仓（正数为多头，负数为空头）。该字段在以下时机更新：

```python
def on_trade(self, trade: TradeData):
    if trade.direction == Direction.LONG:
        self.pos += trade.volume     # 买入 → 持仓增加
    else:
        self.pos -= trade.volume     # 卖出 → 持仓减少
```

**启动时持仓同步**

策略重启时，`CtaEngine` 从 `.vntrader/` 下的持久化文件恢复策略的 `pos` 等变量，避免因程序重启导致持仓状态丢失，实现**状态无损重启**。

**持仓与委托的一致性保证**

`OmsEngine` 维护的 `active_orders` 和 `OffsetConverter` 中的冻结量共同构成持仓状态的完整视图：
- 活动委托的预期成交量被"冻结"，防止重复下单
- 实际成交后更新 `long_td/short_td` 今仓数量
- 非交易时段的持仓数据来自网关的 `query_position()` 查询结果

### 7.6 无 UI 守护进程模式（生产实盘）

生产环境推荐使用**父子进程守护模式**，实现自动故障恢复和交易时段管理：

```python
# examples/no_ui/run.py — 完整守护进程架构

# 父进程：监控交易时段，管理子进程生命周期
def run_parent():
    child_process = None
    while True:
        trading = check_trading_period()

        if trading and child_process is None:
            # 交易时段开始 → 启动子进程
            child_process = multiprocessing.Process(target=run_child)
            child_process.start()

        if not trading and child_process is not None:
            if not child_process.is_alive():
                # 非交易时段子进程退出 → 清理句柄
                child_process = None

        sleep(5)

# 子进程：完整的量化交易引擎
def run_child():
    event_engine = EventEngine()
    main_engine = MainEngine(event_engine)
    main_engine.add_gateway(CtpGateway)
    cta_engine = main_engine.add_app(CtaStrategyApp)

    main_engine.connect(ctp_setting, "CTP")
    sleep(10)                           # 等待连接完成

    cta_engine.init_engine()
    cta_engine.init_all_strategies()
    sleep(60)                           # 等待策略预热
    cta_engine.start_all_strategies()

    # 主循环：定期检查是否需要退出
    while True:
        sleep(10)
        if not check_trading_period():
            main_engine.close()
            sys.exit(0)
```

**交易时段定义**（中国期货市场）

```python
DAY_START   = time(8, 45)   # 日盘开始
DAY_END     = time(15, 0)   # 日盘结束
NIGHT_START = time(20, 45)  # 夜盘开始
NIGHT_END   = time(2, 45)   # 夜盘结束（次日凌晨）
```

**守护进程优势**：
- 子进程崩溃后，父进程在下一个交易时段自动重新启动
- 非交易时段主动退出，避免资源浪费和意外下单
- 无 Qt 依赖，适合 Linux 服务器无头环境部署

├── manager.py   # BarManager：数据管理，维护 BarData 列表和索引
├── item.py      # CandleItem / VolumeItem / ChartItem 基类
├── axis.py      # DatetimeAxis：自定义时间轴
└── base.py      # 公共常量和基础类型
```

**架构特点**
- `ChartWidget` 采用虚拟化渲染：只绘制当前视口范围内的 K 线，支持百万级数据流畅滚动
- `BarManager` 维护完整数据集，通过 `ix`（整数索引）与 K 线一一对应，支持 O(1) 查找
- 自定义 `ChartItem` 基类允许插件式叠加指标线、信号标记等

## 11. 数据流与控制流

### 11.1 实盘交易流（行情 → 策略 → 委托）

```
交易所/券商 API
    │
    ▼  (Gateway 适配层)
on_tick(TickData)
    │  EventEngine.put(EVENT_TICK)
    ▼
  EventEngine（队列处理）
    ├── OmsEngine.process_tick_event()     → 更新内存状态
    ├── Strategy.on_tick(tick)             → 策略逻辑
    │       │  BarGenerator.update_tick()  → 合成 Bar
    │       │  ArrayManager.update_bar()   → 计算技术指标
    │       │  strategy.buy/sell/short/cover()
    │       │  OffsetConverter（开平仓转换）
    │       │  RiskEngine（前端风控）
    │       │  main_engine.send_order(req)
    │       ▼
    │   MainEngine → Gateway.send_order(req)
    │       ▼
    │   交易所返回 orderid
    │
    └── UI Widgets（行情表格实时刷新）

交易所成交回报
    │  on_trade(TradeData)
    ▼
  EventEngine
    ├── OmsEngine.process_trade_event()    → 更新持仓/成交记录
    ├── Strategy.on_trade(trade)           → 策略更新仓位（pos ±= volume）
    └── UI Widgets（委托/成交/持仓更新）
```

### 11.2 回测流

```
回测引擎
    │  加载历史 BarData（从 database/datafeed）
    │
    ├── 逐 Bar 推送给策略
    │       │  Strategy.on_bar(bar)
    │       │  生成委托
    │       ▼
    │   回测引擎内部撮合（不走 EventEngine/Gateway）
    │       │  _cross_limit_order() / _cross_stop_order()
    │       ▼
    │   成交/拒绝 → Strategy.on_order / on_trade
    │
    └── 生成 BacktestingResult
            ├── 净值曲线（逐日）
            ├── Sharpe Ratio（年化）
            ├── Max Drawdown（最大回撤）
            ├── Calmar Ratio
            ├── RGR（4.3.0 新增）
            └── 交易明细表

参数优化
    ├── BruteForce: ProcessPoolExecutor 并行回测所有参数组合
    └── GA: DEAP 遗传算法迭代优化，共享评估缓存
```

### 11.3 分布式部署流

```
服务进程（vnpy_rpcservice）
    ├── MainEngine（完整交易+行情引擎）
    ├── RpcServer（ZMQ REP on :2014）
    │       注册：get_tick / send_order / cancel_order / ...
    └── RpcServer（ZMQ PUB on :4102）
            推送：EVENT_TICK / EVENT_ORDER / EVENT_TRADE / ...

客户端进程 A（策略进程）         客户端进程 B（监控进程）
    RpcClient                       RpcClient
    client.get_tick(vt_symbol)      callback(topic, data)

### 12.2 策略开发范式（以 CTA 为例）

```python
class MyStrategy(CtaTemplate):
    # 参数声明（支持 UI 可视化编辑和参数优化）
    fast_window = Parameter(10)
    slow_window = Parameter(30)
    
    # 变量声明（运行状态，支持持久化）
    fast_ma = Variable(0.0)
    slow_ma = Variable(0.0)
    
    def on_init(self):
        self.load_bar(10)              # 加载历史数据初始化指标
    
    def on_start(self):
        self.write_log("策略启动")
    
    def on_bar(self, bar: BarData):
        am = self.am
        am.update_bar(bar)
        if not am.inited:
            return
        
        self.fast_ma = am.sma(self.fast_window)
        self.slow_ma = am.sma(self.slow_window)
        
        if self.fast_ma > self.slow_ma:
            self.buy(bar.close_price, 1)
        elif self.fast_ma < self.slow_ma:
            self.sell(bar.close_price, 1)
        
        self.put_event()               # 推送策略状态更新事件
```

## 附录：关键设计决策总结

| 设计决策 | 解决的问题 | 实现方式 |
|----------|------------|----------|
| 事件驱动架构 | 模块间解耦，避免循环依赖 | `EventEngine` 队列 + 发布订阅 |
| 双频道事件命名 | 全局 OMS 聚合 vs 精确策略订阅 | `EVENT_TICK` + `EVENT_TICK.vt_symbol` |
| `vt_*` 统一标识符 | 跨网关唯一标识实体 | `gateway_name.id` 拼接约定 |
| 插件包命名规范 | 生态扩展标准化 | `vnpy_<name>` 命名，`get_database()` 动态加载 |
| 持仓转换器 | 处理中国期货市场今/昨仓规则 | `OffsetConverter` + SHFE/INE 特殊逻辑 |
| 抽象基类接口 | 强制驱动包实现完整接口 | `ABC + @abstractmethod` |
| `spawn` 多进程上下文 | 避免 macOS/Windows fork 问题 | `ProcessPoolExecutor(mp_context=spawn)` |
| 环形缓冲区指标计算 | 固定内存占用，O(1) 更新 | numpy 数组 + 移位追加 |
| ZMQ 双通道 RPC | 命令-响应 + 事件流式推送 | REQ-REP（同步）+ PUB-SUB（异步） |
| 无 UI 运行支持 | 服务器部署，去除 Qt 依赖 | `MainEngine` 不依赖 UI，UI 是可选层 |
| 父子进程守护 | 生产环境容错与交易时段管理 | `multiprocessing.Process` + 时段检测循环 |
| 回测/实盘统一模板 | 策略代码零修改切换两种模式 | `CtaTemplate` 共同基类，驱动层不同 |
